- **QBO writes** go through `qteklink-app/src/lib/qbo/*` (`QboClient`, journal-entry builder, Fault parser).

## 2. QBO posting pipeline (per-DAY bulk-JE model)

Each business day (shop-local) posts as **up to three bulk journal entries** ‚Äî never per-RO/per-payment
JEs. Built pure in `qteklink-app/src/lib/daily/daily-je-builder.ts`:

| Category | DocNumber | Contents |
|---|---|---|
| `sales` | `QTL-RO-<date>` | Itemized `Dr A/R` per RO (`desc "RO <number>"`) + credits **aggregated per account** (income / sales tax / PTAL; `desc "Daily sales <date>"`). Discounts netted into income (C5). A $0 RO contributes no lines and is not a constituent. |
| `payments` | `QTL-PAY-<date>` | Every payment's `part:"gross"` lines, itemized both sides in payment order, **never netted** (deposit vs contra routing + refund flips preserved verbatim). |
| `fees` | `QTL-FEE-<date>` | Every payment's `part:"fee"` lines (`Dr CC-Fees / Cr Undeposited`), itemized per payment. Split is **structural** via the `part` tag ‚Äî no line/description matching. |

`DAILY_LINE_CAP = 900` (a bigger category ‚Üí `overLineCap` review item, never posted). DocNumber ‚â§ 18 chars.
An empty category returns `null` (never an empty JE). An unbalanced input propagates `balanced:false` (fail closed).

### 2.1 Live-on-view + nightly cron

Every page view makes the day current, and a nightly cron is the verification net:

- **Live-on-view** (`daily-snapshot.ts` ‚Üí `reconcileDayForView` in `daily-reconcile.ts`): a view first
  refreshes the payment projection (`reduceShopPaymentState`) then **re-reconciles the viewed day**
  (`runDailyReconciliation`) ‚Äî unless the day is terminal (`isDayTerminal`: all rows acknowledged/rejected,
  i.e. Accounting Link's history ‚Äî never re-reconciled, never grows review items). The reconcile returns
  its own `DayBuild` so the read model renders from the same drafts (no double build).
- **Nightly cron** (`nightly-sync.ts` ‚Üí `runNightlySync`): payment-state reduce ‚Üí reconcile ‚Üí payroll
  mirror ingest ‚Üí optional auto-post (only if `auto_post` ON; default OFF) ‚Üí **posted-day sweep** ‚Üí 2-API
  completeness safety net. Each step isolated in its own try/catch so one failure can't discard the others.

### 2.2 reconcile ‚Üí enqueue ‚Üí approve ‚Üí post

```
buildDayDrafts ‚Üí rollupDay (¬ß8 gates)        [daily-reconcile.ts / reconcile/*]
    ‚Üí enqueueDailyPosting  (desired-vs-posted DIFF, per category)   [daily-postings.ts]
    ‚Üí planApproveDay / executeApproveDay  (Pattern S, admin-gated)  [approve-post-day.ts]
    ‚Üí postDailyPostingById (claim + lease + staleness recheck ‚Üí QBO write)  [daily-poster.ts]
```

**Reconcile** (`runDailyReconciliation`, `daily-reconcile.ts`): builds every sale + payment draft, runs
the ¬ß8 gates, persists a ¬ß9 review item per non-postable draft, combines postable drafts into the ‚â§3
category JEs, and enqueues each via the diff. NO QBO write here. Also runs late-payment redate sync
(¬ß4.2) and review-item convergence (¬ß4.1).

**Enqueue diff** (`enqueueDailyPosting`, `daily-postings.ts`) decides per category against the day's LIVE
QBO state; "live JE" = the latest POSTED version's JE unless that version was a delete. Actions:

| `DailyEnqueueAction` | When |
|---|---|
| `new` | fresh version (create v1, or vN+1 after posted-delete / failed / rejected / accepted / correction) |
| `refreshed` / `exists` | a PENDING slot absorbs new content in place (or already matches) |
| `skip` | posted + unchanged, acknowledged (terminal), unchanged-since-terminal, or cosmetic-only delta |
| `frozen` | approved/posting version in flight ‚Äî the poster's claim-time recheck owns divergence |
| `blocked` | unbalanced / over-cap ‚Äî never enqueued (the gate raised the review item) |
| `withdrawn` | pending version for a now-empty day with no live JE ‚Äî system-rejected |
| `obsoleted` | a stuck pending/failed correction became moot (matches live, or descriptions-only) ‚Äî system-rejected |
| `noop` | nothing desired, nothing live, nothing pending |

**Approve** (`approve-post-day.ts`, Pattern S): `planApproveDay` (dry-run) returns the exact per-category
writes (create / full-replacement update / delete) + a `scopeHash` bound to each category's desired source
hash ‚Äî **no writes**. `executeApproveDay` re-derives, rejects if the hash moved (`scope_changed`), then per
category enqueue‚Üíapprove‚Üípost. Scopes: `day` = all three, `sale` = sales JE, `payment` = payments+fees.
Categories post sequentially but independently (QBO writes aren't transactional). Admin-gated in
`src/actions/approve-day.ts` (both branches require `role === "admin"`).

**Post** (`postDailyPostingById`, `daily-poster.ts`): requeue expired leases (`LEASE_SECONDS = 120`) ‚Üí
claim by id (`SKIP LOCKED`) ‚Üí **staleness recheck** (rebuild desired JE, compare hashes; a mismatch
releases the row back to pending with fresh content = re-approval required) ‚Üí by action:
- `create` ‚Üí `POST journalentry` (stable requestid; QBO dedups)
- `update` ‚Üí **full-replacement** re-send under the live JE's Id + current SyncToken
- `delete` ‚Üí `?operation=delete` with `{Id, SyncToken}` (the category emptied)
‚Üí `qteklink_mark_daily_posted` records the QBO id + new SyncToken. update/delete read the live Id +
SyncToken from the latest POSTED row (first-class columns) and **fail closed** on a missing target or
SyncToken (never a blind create/guess).

### 2.3 Idempotent rotating `requestid`, keyed on `sourceStateHash`

`dailyRequestIdFor` (`daily-postings.ts`) = `qtl-` + first 40 hex of
`sha256(shop:realm:day:category:vN:sourceHash)` (‚â§ 44 chars). The **only** function surviving in
`qteklink-app/src/lib/dal/postings.ts` is `sourceStateHash` (canonical key-sorted-JSON sha256) ‚Äî the
staleness fingerprint used by the diff (`dailySourceState`), the approve scope, and the poster's recheck.
Same content ‚Üí same id (crash-safe resend; QBO dedups); **changed content ‚Üí new id** (re-sending changed
content under the old id would return the ORIGINAL response and silently diverge the books ‚Äî audit 2026-06-12).
The `dailyPrivateNoteMarker` is **ledger-internal only** (`proposed_je.marker`) ‚Äî NOT sent to QBO as
PrivateNote: the bank-deposit screen renders the JE-level memo for undeposited rows and only falls back to
per-line descriptions when it's absent (verified 2026-07-09), so the poster omits it (`daily-poster.ts`).

### 2.4 Deposit-lock (6540) ‚Üí non-retryable review item

`qbo/errors.ts` `CODE_TO_KIND` maps Fault code `6540` ‚Üí `deposit_locked`: once a day's Undeposited lines
are swept into a QBO **deposit**, QBO blocks ANY update (full or sparse) until unlinked. Only the
`payments` + `fees` JEs can hit this (they touch Undeposited Funds); the `sales` JE never does.
`classifyPostError` (`daily-poster.ts`) maps it to a **non-retryable** `qbo_deposit_locked` review item;
because the failed version keeps its unchanged hash, the next diff returns `skip` and never re-hammers QBO.
Other classifications: throttle/network ‚Üí retry (back to approved); reconnect/auth ‚Üí `reconnect_required`;
validation mentioning "entity" ‚Üí `ar_entity_rejected`; anything else ‚Üí `qbo_error`.

## 3. Posting roles & the correction model

### 3.1 The 9 posting roles

`qteklink-app/src/lib/mappings/catalog.ts` ‚Äî `POSTING_ROLES`:

```
income ¬∑ sales_tax_payable ¬∑ tire_fee_payable ¬∑ accounts_receivable ¬∑ undeposited_funds
cc_fee ¬∑ noncash_contra ¬∑ store_credit ¬∑ fee_expense
```

- Roles are **derived server-side** (`derivePostingRole(kind, sourceKey)`) ‚Äî never trusted from the client.
  Income-bearing kinds (labor/part_category/fee/sublet) ‚Üí `income`; `noncash_payment_type` ‚Üí `noncash_contra`;
  tax keys ‚Üí `sales_tax_payable` / `tire_fee_payable`; system keys ‚Üí `accounts_receivable` /
  `undeposited_funds` / `cc_fee` / `store_credit`.
- **`fee_expense`** (Chris, 2026-07-01): a FEE mapped to an Expense/Other-Expense account books as a
  contra-expense offset; `feePostingRoleForAccountType` maps the account TYPE (resolved server-side from the
  COA) ‚Üí `income` vs `fee_expense`.
- **`store_credit`** routes store-credit issue/redeem to a liability account.
- Mapping kinds: `labor, part_category, fee, sublet, tax, payment_type, noncash_payment_type, system`.
- The DB RPC `qteklink_set_mapping` (via `qteklink_role_accepts_type` + a BEFORE-write trigger) is the
  **authoritative** role‚Üîaccount-type compatibility gate; `catalog.ts` constants are input-validation +
  display only (`mappings.ts` DAL, `app/mappings/`). PTAL (tire fee) = tires √ó $1 owed unconditionally;
  the display `debitTotal` excludes the `tax_offset` line so RO rows tie to Tekmetric `totalSales`
  (`daily-snapshot.ts`).

### 3.2 Posted-day correction sweep

`qteklink-app/src/lib/dal/posted-day-sweep.ts` (`sweepPostedDays`, nightly + after any date-move decision).
For every day with a POSTED JE, over a 35-day lookback:

1. **Detect date moves** first (`detectDateMoves`): an RO unposted and re-posted to a DIFFERENT day ‚Üí
   the date-move queue (both days HELD) + a Date Change Alert email. Nothing changes in QBO yet.
2. **Re-reconcile** the day (holds applied) ‚Äî any remaining diff stages a correction version (update, or
   delete when the category emptied).
3. **Auto-post the correction** (`applyDayCorrections`): approve as `"system (auto-correction)"` + post,
   **UNCONDITIONALLY ‚Äî there is no `auto_post` gate here**. `auto_post` gates only FIRST-TIME posting
   (`nightly-sync.ts`); a never-posted category is left for the human on the Daily Approvals page.
4. **Email** the Day Correction Alert list ONE consolidated per-day message (`describeDayCorrections`),
   with DB ids resolved to human RO numbers (`resolveConstituentLabels` ‚Äî never a raw id in an email).
   **Same-day churn** (sales only: the RO last changed in Tekmetric on the business day itself) posts
   quietly with no alert (Chris's rule); a later-day change or any non-sales correction sends it.

Change classification is shared with the diff's cosmetic suppression via `@/lib/daily/je-delta`
(`deleted` / `membership` / `descriptions-only` / `amounts`).

## 4. Resolution workflow (self-serve fix-it)

Shipped 2026-07-02 (migration `20260702003000`). The `qteklink_daily_postings.status` enum has **9 values**:
`pending, approved, posting, posted, failed, rejected, needs_resolution, acknowledged, accepted`. Any new
terminal status must be widened **lock-step** in `statusToColumn` (`daily-snapshot.ts`) and the diff.

### 4.1 Failed-state exits (Pattern S)

`qteklink-app/src/lib/dal/failed-posting-resolution.ts` ‚Äî two human exits from a FAILED posting, each with a
dry-run summary + bound scope hash ‚Üí execute:

- **Retry** (`qteklink_retry_daily_posting`, failed‚Üíapproved‚Üípost): "I unlinked the deposit ‚Äî retry now."
  Same version, same content, same requestid (crash-safe resend contract; QBO's requestid dedup returns the
  original if it actually landed). The poster's claim-time staleness recheck stays the last guard.
- **Accept variance** (`qteklink_accept_daily_variance`, failed‚Üí`accepted`, terminal): "Keep QuickBooks
  as-is." The day stops counting it as needs-attention; a later REAL source change still stages v(N+1).

Both, on success, system-close the paired poster review items for `${date}:${category}`
(`qbo_deposit_locked` / `qbo_error` / `ar_entity_rejected` / `reconnect_required`). UI:
`app/approvals/FailedPostingControls.tsx` + `src/actions/failed-postings.ts`.

### 4.2 Late-payment redates

`qteklink-app/src/lib/dal/payment-redates.ts` (`qteklink_payment_redates`, the payment analog of date-moves).
A non-manual payment landing on a day whose payments JE is already posted (would hit 6540) is **HELD out** of
the posted day's desired state (no correction, no 6540); ONE email goes to the Date Change Alert list ("Void
this payment: $X on RO #### ‚Äî take it on a different day"). Lifecycle `pending ‚Üí approved ‚Üí resolved`:
`approved` = admin escape hatch "post it to this day anyway" (hold lifts); `resolved` = auto-detected when the
payment is voided / re-dated in Tekmetric. Detection + auto-resolution run inside the reconcile
(`syncPaymentRedates`), so the queue converges without anyone clicking.

### 4.3 Cosmetic / moot suppression

`enqueueDailyPosting` + `computeScope` short-circuit BEFORE staging when the desired state matches the live
posted JE exactly (hash) or differs ONLY in line-description text (`isCosmeticDelta`, `@/lib/daily/je-delta`):
QBO rejects any update to a deposited JE and a wording-only change matters to nobody. A stuck pending/failed
correction that became moot is system-rejected (`obsoleted`) so the day converges instead of wedging
(the 2026-06-22/26/29 incidents).

### 4.4 One day-scoped attention list + review-item convergence

`assembleDayAttention` ‚Üí `getDailySnapshot().attention` (`daily-snapshot.ts`): the approve lock
(`blockingCount`), the banner, and `/approvals/review` render the SAME list, so they can never disagree.
Review items auto-resolve on convergence (`convergeReviewItems` in `daily-reconcile.ts`): GATE kinds
(re-emitted every reconcile ‚Äî `unmapped, tax_identity, unbalanced, ‚Ä¶`) clear when not re-emitted for a subject
the build can see; POSTER kinds (`qbo_deposit_locked, qbo_error, ar_entity_rejected, reconnect_required`) clear
when the category's latest version leaves `failed`.

### 4.5 "Approved and posted by {user}" panel

`app/approvals/page.tsx` renders a posted-day panel with `deriveApprovedStamp` (`@/lib/approvals/day-status`):
the **human** attribution survives system auto-corrections. The Approve+post button returns only when a
pending correction awaits (`hasPendingCorrection`) ‚Äî otherwise the day shows "Approved and posted on {date}
by {user}". `statusToColumn` maps `posted / acknowledged / accepted` ‚Üí Done; `failed / rejected /
needs_resolution` (and any unknown) ‚Üí Needs attention; `approved / posting` ‚Üí In progress.

> Note: `/postings` is now the **date-move queue** (`app/postings/`, `date-moves.ts`), not a per-RO posting
> ledger (see ¬ß9).

## 5. Acknowledge-day flow ("covered by Accounting Link") ‚Äî LIVE

`acknowledgeDay` (`qteklink-app/src/lib/dal/daily-reconcile.ts`) marks a whole business day approved
**WITHOUT posting** ‚Äî the day is already in QBO via the legacy Accounting Link, so QTekLink records it done
and never posts or corrects it. It reconciles first (so the ‚â§3 category rows exist), **refuses** if any row
is `posted`/`posting`/`approved` (acknowledging would orphan real QBO entries ‚Üí `already_posted`), then flips
every `pending` row to `acknowledged` (terminal via `acknowledgeDailyPosting` ‚Üí `qteklink_acknowledge_daily_posting`).

UI (LIVE): `app/approvals/page.tsx` renders `AcknowledgeDayButton` (`app/approvals/AcknowledgeDayButton.tsx`)
for admins on a not-yet-posted, not-acknowledged day, gated behind a `ConfirmDialog` ("this can't be undone"),
wired to `src/actions/acknowledge-day.ts`. An acknowledged day shows the emerald "covered by Accounting Link"
banner and is excluded from all reconcile/correction paths (`isDayTerminal`, and the diff's `acknowledged`
skip branch).

## 6. Payroll (DONE ‚Äî replaced the Excel pay sheets)

Public entrypoint `qteklink-app/src/lib/dal/payroll.ts`, split across `payroll-*.ts` (~500-line policy);
UI under `app/payroll/`. Roster starts empty until live; completed runs are immutable.

- **Posted-only basis** (decision #51): billed hours AND sales count **only when posted** in Tekmetric ‚Äî no
  completed-status fallback (`payroll-compute*.ts`, `@/lib/payroll/derive`). An unposted RO counts nowhere.
- **Authorized-jobs filter** (#20): every mirror rollup filters jobs `authorized IS TRUE` (declined labor is
  excluded; `selected` is the wrong flag).
- **GP via QBO 6010** (`payroll-compute-gp.ts` `resolveMonthGp`): primary `gp_with_fees = monthSales(incl fees)
  ‚àí partsCost(#37) ‚àí QBO P&L COGS "6010 Technicians"` (`qboMonthTechnicianCostCents`, `src/lib/qbo/reports.ts`).
  The only sanctioned fallback (prorated labor over overlapping runs) fires **only when the QBO fetch throws**
  (Sentry-captured) ‚Äî never a silent 0.
- **Rolling-26 leave rate** (`payroll-leave-rate.ts`): PTO/Holiday/Bereavement pay at the arithmetic MEAN of
  per-period `(base+OT+billed+efficiency) √∑ (reg+OT hrs)` rates over a rolling **26** periods
  (`LEAVE_RATE_WINDOW = 26`, `FETCH_RUNS = 52`), merging frozen completed-run snapshots with pre-QTekLink seed
  history (a completed run wins over a same-period seed). Precedence: `override ‚Üí history ‚Üí seed ‚Üí current_run
  ‚Üí base_rate`.
- **PTO ledger** (`payroll-pto.ts`, `qteklink_pto_ledger`): accrual / usage / rollover-forfeit; `full_time`
  BOOLEAN gates **accrual only** (part-timers still record usage). Tenure tiers + rollover cap from settings.
- **Immutable completed runs**: void-and-clone is the only path (`voidPayrollRun` ‚Üí clones inputs into a new
  open run; full record + `void_clone` alert email).
- **Pattern S confirm** (`completePayrollRun` / `voidPayrollRun`): server-side RPC dry-run ‚Üí `state_hash` ‚Üí
  single-use token bound to it ‚Üí confirm; the RPC recomputes the hash in-transaction and aborts on drift. The
  completion snapshot is **server-computed fresh** (never the live display cache); PTO ledger rows ride INTO
  the confirm RPC (`p_pto_entries`) so they're atomic with the status flip. Pay-summary + negative-balance
  emails fan out post-response via Next 15 `after()` (sequential, never-throw).
- **Webhook mirror-apply** (`payroll-live.ts` `applyMirrorEventsAndRecompute`, route
  `app/api/payroll/mirror-apply`): applies FULL-RO webhook payloads only (partial payloads are skipped;
  recency-guarded vs the mirror so a late older payload can't regress it), marks open runs stale, recomputes
  debounced (`LIVE_RECOMPUTE_DEBOUNCE_MS = 60_000`) reusing the < 6h QBO tech-cost memo. Open runs read
  through a stored `live_snapshot` display cache (recompute-on-stale, `CALC_VERSION`-invalidated);
  completed/voided runs render exclusively from their frozen `snapshot`.
- **Roles**: `viewer < approver < admin` (`QtekRole`); payroll mutations are gated in the actions layer.

## 8. Auth

- **`requireQtekUser()`** (`qteklink-app/src/lib/auth.ts`) ‚Äî server guard on every protected page / action /
  route, three-layer defense:
  1. Microsoft Entra single-tenant restriction (only `jeffsautomotive.com` can complete OAuth);
  2. In-app allowlist `qteklink_allowed_users` (explicit people, keyed on the immutable Entra `oid`) ‚Äî
     narrower than admin-app's whole-domain rule;
  3. `active` flag (a listed-but-deactivated user is rejected distinctly).
  The `oid` is resolved **server-side** inside `qteklink_resolve_allowed_user` from `auth.identities`
  (provider-managed, not user-writable) keyed on the `getUser()`-validated `user.id` ‚Äî never from
  client-writable `user_metadata`. Lookup errors throw ‚Üí **fail closed**. Returns
  `{ email, userId, objectId, shopId, role }` with `role: "viewer" | "approver" | "admin"`; `shopId` is
  server-derived (never from URL/form/client). On reject it redirects to `/login` (carrying a safe `?next=`).
- **`wrapQtekAction(name, inner, tags)`** (`qteklink-app/src/lib/instrument-action.ts`) ‚Äî wraps a Server
  Action in `Sentry.withServerActionInstrumentation` (`recordResponse: false` ‚Äî QBO/Tekmetric payloads carry
  PII) with tags `qteklink_action`, `actor_email`, `actor_object_id`, `qbo_realm_id`. It is **observability
  only** ‚Äî it does NOT call `requireQtekUser`; each action must call it FIRST itself.

> Practical note: writes are gated on `role === "admin"` (e.g. `src/actions/approve-day.ts`); the `approver`
> tier exists in the type but is not a distinct gate on the QBO-posting path. **This is intentional, not dead
> code** ‚Äî qteklink is admin-only today, and the `approver` role is forward-plumbing for the planned
> convergence of all apps into one app with a unified permissions model (Chris, 2026-07-23; "much farther down
> the line, maybe not at all"). Leave it in place; do NOT prune it. See `.claude/memory/app-convergence-roadmap`.

Migration `supabase/migrations/20260723160000_drop_qteklink_retired_posting.sql` tore down the **retired
per-RO posting queue**, fully superseded by the per-day bulk-JE model:

- **Dropped tables:** `qteklink_postings`, `qteklink_ro_state`.
- **Dropped functions:** `qteklink_enqueue_posting`, `qteklink_approve_posting`, `qteklink_reject_posting`,

  stale comments + generated types referenced them); no cron scheduled them; leaf tables (no inbound FK).
- **What remains** in `qteklink-app/src/lib/dal/postings.ts`: only `sourceStateHash` (the canonical staleness
  fingerprint used by the daily diff, approve scope, and poster recheck). `/postings` is now the **date-move
  queue**, not a per-RO ledger.
- **KEPT live (do NOT drop):** `qteklink_settings` + `qteklink_upsert_settings` (back-office / notify /
  payroll / settings DALs), `qteklink_ros` (RO-number cache), `qteklink_daily_postings` (the live per-day
  ledger).
