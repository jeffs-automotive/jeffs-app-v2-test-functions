## 1. Overview

**What it is.** A Next.js 15 / React 19 App Router dashboard for Jeff's Automotive staff, deployed at
`admin.jeffsautomotive.com` (Vercel project `admin-app`). Three operational surfaces plus a landing page:

| Surface | Route | Purpose |
|---|---|---|
| Key tags | `/keytags` | Live key-tag board + assign/release/revert/posted/reconcile + manual reviews + audit |
| Scheduler config | `/schedulerconfig` | Live edit of the scheduler catalog, capacity, appointment types, customer messages |
| Back office | `/back-office` | Service-advisor queue for vendor-bill / re-post issues raised by the office manager |
| Dashboard | `/dashboard` | Landing directory of the surfaces above |

**Stack** (`admin-app/package.json`): Next 15 (`^15.0.0`), React 19, `@base-ui/react` ^1.5 + `shadcn` ^4.8
(component idiom), Tailwind v4 (`@tailwindcss/postcss` ^4), `next-themes` ^0.4 (dark mode), `sonner` ^2
(toasts), `@sentry/nextjs` ^10.52, `@supabase/ssr` ^0.10 + `@supabase/supabase-js` ^2, `zod` ^4.1,
`lucide-react` icons. Dev server runs on port **3001** (`npm run dev`).

**Two backend idioms deliberately coexist:**

- **Keytag data is SPLIT** — reads run **in-process** (service-role DAL); writes + `whoIsOnTag` go over
  the **`orchestrator`** edge function (JSON-RPC). See [§4](#4-keytag-subsystem).
- **schedulerconfig + back-office run fully in-process** — direct `SECURITY DEFINER` RPCs / table reads
  through the service-role client, no orchestrator hop. See [§5](#5-schedulerconfig-subsystem) /
  [§6](#6-back-office-subsystem).

This app deliberately does **not** use `next-safe-action`. Its Server Actions are hand-rolled thin
wrappers that call `requireAdmin()` first and are wrapped by `wrapAdminAction()` for Sentry
instrumentation ([§3](#3-auth-model)).

## 3. Auth model

**`requireAdmin()`** — `admin-app/src/lib/auth.ts:34`. The single server-side guard, called at the top of
every protected page **and** every Server Action (never in a layout — Server Actions can be POSTed directly,
bypassing layout gates; `AppShell.tsx` documents this contract).

Two-layer defense:

1. **Microsoft Entra tenant restriction** — configured in the Supabase Auth Azure provider; only the
   `jeffsautomotive.com` M365 tenant can complete OAuth.
2. **Email-suffix check** (`auth.ts:46`) — belt-and-suspenders: `email.toLowerCase().endsWith("@jeffsautomotive.com")`.
   A federated guest that slips past the tenant config is signed out and bounced to
   `/login?error=unauthorized_domain`.

`requireAdmin()` returns `{ email, userId }`; no-session → `redirect("/login")`. `getAdminSession()` is a
soft, non-redirecting variant for conditional rendering. **Single-role** today (per `PLAN.md` D6): anyone
with `@jeffsautomotive.com` gets full access; the audit log captures who-did-what. RBAC is a documented
future extension of this helper.

**`wrapAdminAction()`** — `admin-app/src/lib/instrument-action.ts:38`. This app's substitute for
next-safe-action. It is **observability only** — it does **not** call `requireAdmin()` for you (each action
must do that itself, first). It wraps the inner action in
`Sentry.withServerActionInstrumentation(name, { recordResponse: false }, …)` and sets tags
`admin_action`, and optionally `actor_email` / `orchestrator_tool`. `recordResponse:false` because
returns can carry PII-adjacent customer/vehicle data. It is variadic so it wraps both simple
`(args)=>Promise` actions and `useActionState` `(prev, formData)=>Promise` actions.

**Actor identity** flows as `email` from `requireAdmin()` into every write — for orchestrator calls it is
sent as the `X-Actor-Email` header; for direct RPCs it is passed as `p_actor`. It is **never** read from the
client.

**Supabase clients** (`src/lib/supabase/`):

- `server.ts` — cookie-bound `@supabase/ssr` server client (auth: sign-in/out, `getUser`).
- `admin.ts` — service-role client (`createSupabaseAdminClient`), server-only, bypasses RLS by design;
  powers the read-DAL / write-DAL / back-office / orchestrator bearer.
- `client.ts` — browser client (used by `LoginButton`).
- `resolve-keys.ts` — env resolver: prefers the 2026 `SUPABASE_SECRET_KEYS` JSON-dict form, falls back
  `SUPABASE_SECRET_KEY` → legacy `SUPABASE_SERVICE_ROLE_KEY`; the service-role resolver **never** reads a
  `NEXT_PUBLIC_`/anon var (tested in `tests/unit/resolve-keys.test.ts`).

**shop_id** — `resolveAdminShopId()` (`src/lib/scheduler/shop-id.ts`) resolves server-side:
`SCHEDULER_ADMIN_SHOP_ID` env var, else fallback `7476` (Jeff's sandbox shop). Never from client input.
Single-shop today; multi-shop is a documented deferral.

**Sentry init:** `sentry.server.config.ts` (Node), `sentry.edge.config.ts`, `instrumentation.ts`
(`onRequestError`), `instrumentation-client.ts`. Global tags `surface: admin-app`, `shop_id: 7476`; PII
scrubber in `beforeSend`; `SENTRY_DSN` optional (no-ops if unset).

## 4. Keytag subsystem

The keytag console at `/keytags` has **6 tabs** (`src/components/keytag/KeytagsTabs.tsx`), URL-synced via
`?tab=` and `window.history.replaceState` (so a tab click never re-runs the 6 tab Server Components):

| Tab value | Label | Content component | Data path |
|---|---|---|---|
| `dashboard` | Dashboard | `DashboardTab` | **read** — `getCachedDashboard()` (60s `unstable_cache`) → read-DAL |
| `live` | Board | `LiveBoardTab` → `BoardClient` | **read** — `loadBoardState()`; **writes** via orchestrator actions |
| `posted-revert` | Posted / Revert | `PostedRevertTab` | writes via `markKeytagPosted` / `revertKeytagToAssigned` |
| `reconcile` | Reconcile | `ReconcileTab` | write via `runBulkReconcile` |
| `manual-reviews` | Manual reviews | `ManualReviewsTab` | **read** — `getManualReviews()`; resolve via `resolveManualReview` |
| `audit` | Audit history | `AuditHistoryTab` | **read** — `getAuditHistory()` |

> **Suspense nuance** (`app/keytags/page.tsx` header comment): every tab is wrapped in `<Suspense>` **except
> Live/Board**, which renders directly. A Suspense fallback would unmount the Live subtree on each Server
> Action re-render, destroying an open Pattern-A `ConfirmationDialog` and its already-issued token before
> the user can Confirm (the "B1" force-assign regression). Board reads are fast DB lookups, so the in-place
> re-render is acceptable.

### 4.1 Reads — in-process service-role DAL

`src/lib/keytag/read-dal.ts` (+ `queries/`) replaced the orchestrator HTTP hop for the four READ surfaces.
Each fn builds `createSupabaseAdminClient()`, resolves `shop_id` server-side, calls a pure query under
`queries/`, and wraps the call in a **10s seatbelt** (`withTimeout`) that **throws** on timeout or DB error
(never swallows to empty — callers render an error card):

| Fn | Query module | Feeds |
|---|---|---|
| `getDashboard()` | `queries/keytag-dashboard.ts` | Dashboard tab (via `dashboard-cache.ts`) |
| `getWipKeyTags()` | `queries/wip-keytags.ts` | Live board "tagged" list |
| `getManualReviews(args)` | `queries/manual-reviews.ts` | Manual reviews tab |
| `getAuditHistory(args)` | `queries/audit-history.ts` | Audit history tab |

- `dashboard-cache.ts` wraps `getDashboard()` in `unstable_cache` (`DASHBOARD_TTL_SECONDS = 60`, tag
  `keytag-dashboard`) — a shop-global snapshot, no per-advisor key. The dashboard snapshot is now a **pure
  DB read** (customer names come from denormalized `keytags.customer_name`, not a Tekmetric walk — that 45s
  walk was the 2026-06-25 board-"spin" root cause).
- `load-board-state.ts` composes `getWipKeyTags()` (tagged list) + `getManualReviews({ only_open:true })`
  (the "untagged / needs-a-tag" rows, sourced from open manual reviews of the needs-a-tag categories).

### 4.2 Writes + whoIsOnTag — the `orchestrator` edge function

`src/lib/orchestrator/client.ts` is the typed gateway. **Only these 7 tools go over the wire**, via
`callKeytagTool(...)` (one Server Action file each under `src/actions/keytag/`):

| Server Action file | Orchestrator tool | Kind |
|---|---|---|
| `assign-keytag.ts` | `assignKeytagToRo` | write (Pattern A when force-assigning) |
| `release-keytag.ts` | `releaseKeytagFromRo` | write (Pattern A when `posted_ar`) |
| `revert-keytag.ts` | `revertKeytagToAssigned` | write (Pattern A when `posted_ar`) |
| `mark-keytag-posted.ts` | `markKeytagPosted` | write (Pattern A always) |
| `run-bulk-reconcile.ts` | `runBulkReconcile` | write (no Pattern A) |
| `resolve-manual-review.ts` | `resolveManualReview` | write |
| `who-is-on-tag.ts` | `whoIsOnTag` | **read** (kept on the gateway) |

> **Count:** that is **6 mutating tools + `whoIsOnTag` (a read) = 7 orchestrator-routed tools.** Project
> memory phrases this as "7 writes + whoIsOnTag", which over-counts writes by one — `whoIsOnTag` is a read.
> A `board-state.ts` action also exists but it wraps the in-process `loadBoardState()`, not an orchestrator
> call.

**Transport contract** (`callOrchestratorRpc`, `client.ts:147`):

- **Wire format:** JSON-RPC 2.0, method **`tools/call`** — the *only* method the client ever invokes
  (`{ jsonrpc, id, method:"tools/call", params:{ name, arguments } }`). Response is
  `{ content:[{type:"text", text:<JSON-string>}], isError }` → unwrapped `JSON.parse`.
- **Auth:** `Authorization: Bearer <service-role>` + `apikey: <service-role>` + `X-Actor-Email: <email>`
  (the SERVICE_ROLE + X-Actor-Email branch, server `SERVER_VERSION` 0.4.0).
- **Host allowlist (fail-closed, `buildOrchestratorUrl`):** URL is derived from the configured Supabase URL
  as `…/functions/v1/orchestrator`. **Layer 1** — derived host MUST end with `.supabase.co`. **Layer 2** —
  derived host MUST exactly equal the project's own Supabase URL host. Either failure throws before any
  bearer is sent (tested in `tests/unit/orchestrator-host-allowlist.test.ts`).
- **Timeout:** default 30s (`AbortSignal.timeout`); bump for `runBulkReconcile` (5–30s Tekmetric walk).
- **Errors:** network / non-2xx / JSON-RPC error / `isError` all fold to `Sentry.captureMessage`/
  `captureException` + a structured `OrchestratorClientError`. Tool-level `ok:false` (not-found, validation,
  Pattern-A confirmation prompts) is a normal return, not an exception.

**Typed tool map** — `src/lib/orchestrator/types.ts` mirrors the edge Zod schemas (redeclared, not imported,
because the edge sources use Deno `npm:` imports). `KeytagToolMap` declares **11** tool shapes for typing;
only the 7 above are actually dispatched through the client (the other 4 —
`getKeytagDashboard`, `listWipKeyTags`, `listManualReviews`, `getKeytagAuditHistory` — are typed there but
served in-process by the read-DAL). Key exports: the `ConfirmationRequiredResult` **Pattern A** envelope
(`{ ok:false, needs_confirmation:true, confirmation:{ token_id, expires_at, action_kind, scope_summary } }`),
per-tool arg/result unions, and the `isConfirmationRequired(v)` type-narrow used by the UI to switch into
the confirmation-modal flow.

**Pattern A confirmation (UUID token) in the UI** — for the mutating tools that can cause damage, the first
call returns `needs_confirmation` + `scope_summary` + a token; the user reads the exact scope in
`ConfirmationDialog.tsx` and clicks Confirm in a **new turn**, re-calling with `confirmation_token`. Server
gates expiry (the client keeps Confirm clickable even past the shown expiry — `ConfirmationDialog` test L1).
Successful writes intentionally **do not** `revalidatePath("/keytags")` (the board reconverges optimistically
+ via `LiveBoardPoller`); page-wide revalidation was the post-success "continually loads" spin
(2026-06-24 fix), guarded by `tests/unit/keytag-actions-revalidate.test.ts`.

## 5. schedulerconfig subsystem

`/schedulerconfig` (`app/schedulerconfig/page.tsx`) is the **direct-webforms** rewrite (2026-07-02). The
older MD-upload / Pattern-S / orchestrator admin pipeline is **retired** — every tab renders LIVE table data
and mutates through direct `SECURITY DEFINER` RPCs that commit the change + its audit row atomically with
`updated_at` staleness protection. The RSC fetches all tab data in **one** `Promise.all` round; mutations
`revalidatePath` + `router.refresh()` re-run the fetch.

**11 tabs** (`src/components/scheduler/direct/DirectConfigTabs.tsx`, in order):

| # | Key | Label | Tab component |
|---|---|---|---|
| 1 | `services` | Services | `ServicesDirectTab` (routine + testing) |
| 2 | `types` | Appointment Types | `TypesDirectTab` |
| 3 | `templates` | Messages | `TemplatesDirectTab` |
| 4 | `cardtext` | Card Text | `CardTextDirectTab` (+ Heritage card preview) |
| 5 | `subcategories` | Subcategories | `SubcategoriesDirectTab` |
| 6 | `questions` | Questions | `QuestionsDirectTab` |
| 7 | `guidelines` | Guidelines | `GuidelinesDirectTab` |
| 8 | `limits` | Limits | `LimitsDirectTab` |
| 9 | `closeddates` | Closed Dates | `ClosedDatesDirectTab` (closed dates + blocks) |
| 10 | `operations` | Operations | `OperationsDirectTab` (orphan finder + sync) |
| 11 | `history` | History | `RecentChangesList` (audit log) |

**Reads** — `src/lib/scheduler/read-dal.ts` (server-only, service-role, `shop_id`-scoped). One reader per
tab: `listRoutineServices`, `listTestingServices`, `listSubcategories`, `listQuestions`, `listGuidelines`,
`listAppointmentLimits`, `listClosedDates`, `listAppointmentBlocks`, `listAppointmentTypes`,
`listMessageTemplates`, `listCardText`, `findOrphans`, and `listAuditLog` (via the existing outer RPC

**Writes** — `src/lib/scheduler/write-dal.ts` → thin actions in `src/actions/scheduler/`:

  `updateSubcategoryEnrichment`, `updateSubcategoryServiceMap`, `updateQuestionRequiredFacts`,
  `upsertConcernQuestion`, `updateCategoryGuideline`.
- `direct-config-actions.ts` — `setAppointmentLimits`, `add`/`removeClosedDate`,
  `block`/`unblockCapacity` (direct `appointment_blocks` table writes), `set`/`deactivateAppointmentType`,
  `setMessageTemplate`, `runAppointmentsSyncDirect` (invokes the `appointments-sync` edge fn),
  `set`/`resetCardText`.

Every write RPC takes `p_shop_id` (server-resolved), `p_actor` (from `requireAdmin`), and
`p_expected_updated_at` for optimistic concurrency; `write-dal.ts` returns
`{ ok:true, id?, updated_at? } | { ok:false, code:"stale_write"|"rpc_error", error }`. All actions are
`wrapAdminAction`-wrapped and call `requireAdmin()` first.

Supporting client libs: `template-renderer.ts` (+ `.test.ts`), `card-merge-fields.ts`,
`direct-form-state.ts`, `types.ts`, plus the Heritage preview under
`src/components/scheduler/direct/` (`HeritageCardPreview.tsx`, `card-preview.css`, `card-preview-manifest.ts`)
— Poppins is published in `layout.tsx` **only** to render this card-text preview in the customer's exact type.

## 6. back-office subsystem

`/back-office` (`app/back-office/page.tsx`) is the **service-advisor** half of the cross-app back-office
module (the office manager's half lives in `qteklink-app`). Both read/write the shared `back_office_issues`
table + RPCs. Two sections: **"Needs your fix"** (`status = sent_to_sa`) and **"Submitted — waiting to
verify"** (`status = awaiting_verify`).

**DAL** — `src/lib/back-office.ts` (service-role, `shop_id` server-resolved):

- `listSaQueue(shopId)` — pulls `sent_to_sa` + `awaiting_verify` rows, mapped to `SaQueueIssue`.
- `submitFix(shopId, issueId, actor, saNote)` — RPC `back_office_submit_fix` (`sent_to_sa → awaiting_verify`).
- `notifyBackOffice(...)` — fires the `back-office-notify` edge fn after submit; never throws into the
  caller (logs to Sentry on failure).

**Actions** — `src/actions/back-office/submit-fix.ts` and `fetch-attachments.ts`. Components:
`SaQueueRow`, `IssueDetailDialog`, `ReopenedHistory`, `ViewAttachmentButton`, `status.tsx`.

**No QBO client.** admin-app has **no** Intuit/QBO integration code. To let advisors view the same
parts-invoice image the office manager sees, `fetchAttachmentsAction` (`fetch-attachments.ts`) **proxies**
to qteklink-app's internal endpoint: `POST {QTEKLINK_INTERNAL_URL}/api/back-office/attachment` with the
service key + server-resolved `shop_id` (qteklink owns the shared QBO token lifecycle). Read-only; degrades
to a friendly message on error. Only `Bill`/`Purchase` txn types are accepted; the returned QBO URL is
short-lived so `ViewAttachmentButton` re-mounts fresh each open.

> Note: `intuit-oauth` is still listed in `admin-app/package.json` but has **zero** import in `src/`/`app/`
> — a leftover dependency, not a live QBO client (see [§10](#10-known-discrepancies)).

## 7. UI / styling

**Component system.** shadcn (`shadcn` ^4.8) built on **`@base-ui/react`** ^1.5 — the primitives under
`src/components/ui/` (`button`, `badge`, `dialog`, `input`, `label`, `separator`, `tabs`, plus non-base-ui

**Tokens** (`app/globals.css`, Tailwind v4 `@theme` + `:root`/`.dark`):

## 9. Recent changes (2026-07-23)

Three landings on 2026-07-23, all reflected in the code above:

1. **Orchestrator rename: `orchestrator-mcp` → `orchestrator`.** The edge function was renamed; the
   client's `ORCHESTRATOR_FUNCTION_NAME = "orchestrator"` (`src/lib/orchestrator/client.ts:32`) and the
   derived URL is `…/functions/v1/orchestrator`. Context (per memory): the Claude-Desktop-era `mcp-auth`
   function and the OAuth token tables/functions were dropped (migration `20260723150000`); the
