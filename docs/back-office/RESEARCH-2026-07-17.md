### ⚠️ Stale/wrong in our own `docs/qbo/` + `quickbooks` skill

| Item | Our docs say | Current reality | Source |
|---|---|---|---|
| **Refresh-token lifetime** | 100-day rolling expiry | **100-day rule REMOVED** → 5-year hard max, still rotates ~24h; new hard-expiry field in token response; **"Reconnect URL" now mandatory** in app settings | [Intuit blog 2025-11-12](https://medium.com/intuitdev/important-changes-to-refresh-token-policy-8443779d40db) |
| **Webhook payload** | CloudEvents opt-in; parse classic `eventNotifications` | **CloudEvents becoming MANDATORY**, classic deprecated, **deadline July 31 2026** (~2 weeks out); one notification can carry MULTIPLE companies | [Intuit blog 2025-11-12](https://blogs.intuit.com/2025/11/12/upcoming-change-to-webhooks-payload-structure) |
| **Custom fields on Bills** | desktop-only | NEW Custom Fields API (Dec 2025) adds them to Bill/Purchase — **Advanced/Enterprise tier ONLY** (Jeff's = Essentials → N/A) | [Intuit blog 2025-12-01](https://blogs.intuit.com/2025/12/01/custom-fields-api-extending-quickbooks-online-with-flexible-metadata/) |
| **minorversion** | "SDK default 75, latest unconfirmed" | **Confirmed 75 = default AND floor**; 1–74 deprecated Aug 1 2025 (lower values ignored, served as 75) | [Intuit Medium](https://medium.com/intuitdev/changes-to-our-accounting-api-that-may-impact-your-application-c330bd1a06f5) |

**Signature verification UNCHANGED** and explicitly unaffected by the CloudEvents migration: HMAC-SHA256 over
the raw body, keyed by the app Verifier Token, base64-encoded, compared to `intuit-signature`. Our deployed
`qbo-webhook` already does exactly this.

### Reading a vendor bill by the office manager's "invoice number"

- The office manager types a vendor **invoice number** = the QBO transaction's **`DocNumber`** (e.g.
  `110381`, `01877130133`, `6IV941884`). It's a **string** — leading zeros match literally.
- Query: `SELECT * FROM Bill WHERE DocNumber = '110381'`. But the same doc may have been entered as a
  **`Purchase`** (expense/check/CC) rather than a `Bill` → **must query both entities** (no cross-entity
  OR/JOIN; issue two queries or one batch of two).
- **DocNumber is NOT guaranteed unique.** Uniqueness is only enforced on create if the company enabled the
  duplicate-warning preference; across history / across vendors / across entity types you can get 0, 1, or
  **many** matches. → Treat the result as an array; disambiguate by Vendor + TxnDate + amount before
  showing one. [data-queries](https://developer.intuit.com/app/developer/qbo/docs/learn/explore-the-quickbooks-online-api/data-queries)
- Fields we get back on a Bill: `VendorRef{value,name}`, `DocNumber`, `TxnDate` (bill date), `DueDate`,
  `TotalAmt`, `Balance`, `Line[]` (`AccountBasedExpenseLineDetail` / `ItemBasedExpenseLineDetail`, each with
  `Amount` + `Description` ≤4000 chars), `PrivateNote`, `SyncToken`, `MetaData`.
- **No custom field for RO number available on Essentials.** If an RO # is needed on a Bill without a tier
  upgrade, it must come from `PrivateNote` / `DocNumber` / a line `Description` and be parsed.

### Other facts
- **OAuth:** access token 1h (unchanged); `intuit-oauth` npm still maintained (~4.2.x); accounting scope
  `com.intuit.quickbooks.accounting`.
- **Rate limits:** 500 req/min per realm, max 10 concurrent → HTTP 429 `ThrottleExceeded`. Batch max 30
  (SDK guards 25); batch endpoint ~120 req/min.
- **Sandbox vs prod:** different hosts + companies + tokens; CloudEvents toggle + "send test event" work in
  both, independently; changes land in sandbox ~6-7 weeks before prod.

## 3. Cross-app + workflow precedents

### Auth (the central architecture decision)
- **admin-app = flat gate, NO roles.** `requireAdmin()` (`admin-app/src/lib/auth.ts:34`) only checks
  `email.endsWith("@jeffsautomotive.com")`. Header comment invites adding a role later. `AdminSession` =
  `{ email, userId }` only.
- **qteklink-app = allowlist + roles.** `requireQtekUser()` (`qteklink-app/src/lib/auth.ts:77`) →
  `qteklink_allowed_users` keyed on Entra `oid`, roles `viewer | approver | admin` (`approver` is defined
  but currently unused in the management UI).
- **Shared identity, NOT shared authz.** Both apps use the SAME Supabase project + SAME Entra app
  registration → same `auth.users`/`auth.identities`. No shared app-level user/role table.
- **No shared code package / monorepo.** They share only the DB (migrations + edge functions) and
  hand-copied TS. "Same data + status machine in both apps" ⇒ one shared table + shared RPCs, each app
  builds its own thin DAL/actions/UI (repo idiom = duplication).

### Status / state-machine idiom
- **Zero Postgres ENUM types repo-wide.** Universal idiom = `col TEXT NOT NULL DEFAULT '...' CHECK (col IN
  (...))`, widened via DROP/ADD CONSTRAINT.
- Transitions enforced by **SECURITY DEFINER RPCs, not triggers**; tables are service-role-SELECT-only with
  writes REVOKEd; each transition is a guarded `UPDATE ... WHERE status = '<from>'` (optimistic from-state).
- **Closest existing analog = `qteklink_review_items`** (`20260607070000...`): `status` `open|resolved`,
  one-open-per-subject partial unique index, `detail`/`resolution` JSONB, resolver columns. Recommended base.
- **Audit model = `keytag_audit_log`** (`20260511180000...`): `prior_status`/`new_status`, `source`
  (`claude_desktop|webhook|cron|manual_sql`), `user_label`, `reason`. A ready-made two-actor trail.

### Reopened-RO detection (Chris: "we already track this in the keytag app")
Tekmetric webhooks (token-only auth, no HMAC, order not guaranteed) carry the full RO object. We CAN detect:
- **(a) RO unposted** — explicit `ro_unposted` event, classified + stored ONLY by `qteklink-webhook`
  (`index.ts:79`, regex `^Repair Order #\d+ unposted by`). (The keytag app infers reopen from a later
  `statusId===2`.)
- **(b) total sales changed after posting** — via the daily posted-day sweep diffing `qteklink_events`
  history (`dal/posted-day-sweep.ts`), `classifyDelta changeKind:"amounts"`.
- **(c) posted date changed** — via `detectDateMoves` → `qteklink_ro_date_moves` + Date Change Alert email
  (`dal/date-moves.ts`). Ordering keys on `received_at` because Tekmetric backdates unpost/repost.
- The Notion "Unposted Repair Orders" table's `Name` = `Repair Order #<n> unposted by <user>` matches the
  keytag/qteklink unpost signal.

### Email + daily digest
- **No React Email / template engine** — all hand-built inline-styled HTML strings; provider = **Resend**.
- qteklink entry point: `sendQteklinkEmail()` (`src/lib/dal/notify.ts`) → `qteklink-email` edge fn → Resend.
  Never throws. **Reuse this.**
- **Daily digest template = `keytag-daily-report`**: cron (`0 11 * * 1-6`) → `scheduler_invoke_edge_function`
  → bearer → shared data-builder → HTML (stat tiles + a **stale section** + "always render header, show
  positive empty state") → Resend with per-day idempotency key. The **shared-data-builder feeds BOTH the
  email and the in-app dashboard tab** (no drift) — copy this trick.
- Recipients today = env vars. For per-shop editable recipient lists, the idiom is `qteklink_settings`
  (single row per `(shop_id, realm_id)`) — comma-separated `text` columns (e.g. `date_change_alert_emails`)
  or a `jsonb` sub-key (payroll pattern). No generic `module_settings` table exists.

### Freshness = polling, not Realtime
Supabase Realtime is used NOWHERE. Pattern = visibility-gated `setInterval(() => router.refresh())`
(admin `DashboardPoller` 60s, qteklink `AutoRefresh` 45s) + immediate `router.refresh()` after a mutation.
"SA sees the item appear" = ~45-60s poll.

### Dashboard/metrics precedents
Keytag Dashboard tab (stat cards + count tables, in-TS `.filter().length`) and qteklink approvals
(KPI cards + per-status roll-up via in-TS reduce). **No month-to-date / open-vs-closed / rolling-period
aggregate exists anywhere** — the "open vs closed this month + stale N days" dashboard is new ground.

## 5. Chris's decisions (2026-07-17)

1. **Build sequence:** whole module in one pass.
2. **Reopened ROs:** reuse the existing qteklink Tekmetric detection pipeline; **status must encode the
   change type** — unposted / reposted / posted-to-different-date / posted-with-total-sales-change / a
   combination (richer than Notion's single "Was Amount Changed?").
3. **SA access:** visible to all admin-app users (no new RBAC).
4. **Parts invoice image:** pull it **from QuickBooks** (not a Supabase Storage upload) → needs the QBO
   Attachable API (query attachments for a Bill, download via TempDownloadUri — attachments are NOT tier-
   gated, unlike custom fields). **Verify at plan/build.**

5. **RO# source (Chris):** "There is a **customer line** in QBO for the RO#." → the RO number lives on a
   bill **line** that references the customer (the line's `CustomerRef` and/or its `Description`), NOT a
   custom field. Our `SELECT * FROM Bill WHERE DocNumber=...` returns `Line[]`, so the fetch can auto-fill
   RO# + vendor + bill date. **Verify the exact line field against a real Jeff's bill during planning.**
6. **QBO tier (Chris):** **Essentials.** No custom fields on bills (consistent — RO# is a line reference,
   not a custom field). Attachments API is not tier-gated, so pulling the parts-invoice image still works.
