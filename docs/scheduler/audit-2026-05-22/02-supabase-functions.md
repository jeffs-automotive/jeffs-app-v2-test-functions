# Supabase functions audit

## Executive summary

Reviewed 18 edge functions + 25 `_shared/` modules + `supabase/config.toml`. Overall posture is strong: scheduler-app surface has consistent Pattern A bearer auth via `_shared/scheduler-auth.ts`, webhook endpoints validate URL `?token=` correctly, the orchestrator-mcp OAuth flow hashes tokens at rest with SHA-256, the manual-review + UUID-confirmation patterns are implemented cleanly, and PII scrubbing in `sentry-edge.ts` is comprehensive (blocklist + regex + value-walker over event payloads).

**Two BLOCKER findings.** Both stem from the same architectural pattern — Tekmetric-side test/probe surfaces (`tekmetric-api-testing`, `tekmetric-bootstrap`) gated only by `verify_jwt = true` (the publishable anon key is accepted). The anon key is by design exposed to browsers, so any third party can in principle reach these functions. `tekmetric-api-testing` exposes write ops (POST/PATCH/DELETE to Tekmetric) gated by a UUID confirmation pattern, but UUID confirmation only forces a two-step flow within the SAME caller — it does not prevent an attacker who possesses the anon key from doing the same two-step. `tekmetric-bootstrap` writes a non-expiring access token into the Vault.

Eleven additional functions deploy without a `verify_jwt` config block (Supabase defaults that to `true` = anon-key-passes) but their handlers DO have additional internal auth checks, so they're safer.

13 of 18 functions lack `withSentryScope` wrap. In the Deno SDK, this means breadcrumbs and tags from one request can surface on errors from a concurrent request in the same isolate — a multi-tenant context-leak risk. For single-shop Phase 1 the practical impact is small (one shop, no cross-tenant data), but the rule says this is required.

Other IMPORTANT findings include non-constant-time webhook token comparison (timing-attack vector is theoretical for Tekmetric webhooks but the pattern is wrong), missing webhook idempotency on the `tekmetric-webhook` ingestion path (writes via `.insert()` not `.upsert(...).onConflict()`), unchecked `.update()` errors in several call sites (sb call returns ignored), and the `tekmetric-bootstrap` flow returning an access-token preview into HTTP response bodies even though it's the first 8 chars only.

## Patterns observed (positive)

- **Pattern A bearer auth via `_shared/scheduler-auth.ts`** is well-designed: constant-time compare (line 127-134), env-var fallback (legacy + new sb_secret_* format + JSON dict format), structured diagnostic on auth failure (first-8-chars prefix; doesn't leak the secret), and a single canonical `RESOLVED_SERVICE_ROLE_KEY` export. The 4 scheduler-* + 2 keytag-* + appointments-sync + transcript-dispatcher functions all use it consistently.
- **OAuth implementation in `mcp-auth` + `orchestrator-mcp` is solid:** SHA-256 hash at rest (never raw tokens in DB), PKCE S256 required, single-use auth codes with race-safe `used_at IS NULL` filter, rotating refresh tokens per OAuth 2.1 §6.1, scope-subset enforcement on refresh (line 555-560), and bound Resource Indicator metadata. Only weakness is I6 (resource not checked at token-use time).
- **Sentry PII scrubbing in `_shared/sentry-edge.ts:121-215`** is comprehensive: 30+ key blocklist, EMAIL/PHONE/OTP regex scrubbers, recursive value walker, applied to `user` / `message` / `exception.value` / `contexts` / `extra` / `breadcrumbs` / `request.{data,headers,query_string}`. Defense-in-depth note in the header about server-side Sentry Data Scrubber rules is good practice.
- **UUID confirmation token pattern** in `_shared/keytag-confirmation.ts` is well-implemented: scope_hash binds token to exact mutation, user_label match enforced, 5-min TTL, atomic consume RPC, no replay across different ROs. Manual review (6-char code) pattern in `_shared/manual-review.ts` is similarly clean — composite dedup gate `(category, ro_id)`, brute-force lockout, lenient input normalization, single-use codes.
- **`appointments-sync` cardinality handling at `appointments-sync/index.ts:265-313`** correctly de-duplicates Tekmetric pagination-overlap via `Map<id, row>` last-write-wins. Defends against Postgres "cannot affect row a second time" upsert error.
- **Webhook auth: `?token=<TEKMETRIC_WEBHOOK_TOKEN>` query param** is the correct pattern given Tekmetric's no-custom-headers limitation. Query string is stripped before persistence (`safeQueryString` in `tekmetric-webhook/index.ts:127-132` and headers via `safeHeaders` line 117-125). Authorization + Cookie + Set-Cookie all filtered from `raw_headers`.
- **`tekmetric-webhook` and `keytag-tekmetric-webhook` always return 200** after attempting work, even on internal errors. Correct webhook semantics — Tekmetric retries on non-200, which is undesirable when the internal failure is our DB outage (retry won't help, retry will just amplify load). The internal error is logged via `console.error` (and would reach Sentry via Log Drain when on Team plan).
- **Resend email idempotency-key per code** in `manual-review-email.ts:57` and `transcript-dispatcher.ts:159` prevents accidental double-send.
