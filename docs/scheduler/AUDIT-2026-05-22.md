> **Pre-launch readiness audit** covering scheduler-app (Next.js wizard), supabase edge functions, DB schema + migrations, Sentry / Vercel / Telnyx / Resend / Anthropic / Tekmetric integrations, observability + secrets, and tests + CI. Six parallel sub-agents + orchestrator MCP probes.
>
> **TL;DR:** code quality is high. Wizard, DAL, edge functions, RLS coverage, SECURITY DEFINER discipline, PII redaction, OAuth flow — all solid. **10 BLOCKERs** concentrated in three areas: (a) 2 dev/test surfaces (`tekmetric-api-testing` + `tekmetric-bootstrap` edge fns) accept the publishable anon key, (b) 2 scratch tables have RLS disabled, (c) test coverage of the highest-risk code paths (3-stage diagnostic LLM, Tekmetric webhooks, run-diagnostics aggregator) is zero, and there's no CI gate at all. **Plus**: 1 cron silently failing daily for 3 days, 1 webhook-idempotency-at-DB-level gap, no Vercel BotID on the unauthenticated chat/OTP endpoints. **Nothing customer-data has leaked** — these are pre-launch fixes, not active incidents.

### Headline assessment

The codebase is **mature, well-instrumented, and well-conventioned**. The 4 most important architectural patterns (thin-action/fat-DAL, `wrapAction` Sentry instrumentation, RLS `deny_all` default, `_shared/scheduler-auth.ts` Pattern A bearer) are followed consistently. Zero raw-PII-in-logs. Zero hardcoded secrets. Zero empty-catch error swallowing. SECURITY DEFINER `search_path` discipline is universal across all 27 functions.

The 10 BLOCKERs cluster in three categories — none are active incidents, all are pre-launch concerns:

1. **Two Tekmetric edge functions (`tekmetric-api-testing`, `tekmetric-bootstrap`) deploy with `verify_jwt = true` default,** which means the Supabase publishable anon key (by design exposed to browsers) is sufficient to invoke them. The UUID confirmation flow in api-testing only forces step1→step2 sequencing within a single caller; it does not gate access. **An attacker with the anon key could POST/PATCH/DELETE arbitrary Tekmetric appointments.** Likely fix: add `verify_jwt = false` in `config.toml` + add `checkSchedulerBearer` at top of handler. ~1 hour.
2. **Two dev/scratch tables (`_bulk_keytag_backfill`, `_smoke_test_run`) have RLS disabled.** Anon role can read/write 149 + 42 rows respectively. Likely no live PII, but they should either get RLS+deny_all OR be dropped. ~15 min.
3. **No CI gate AT ALL** — `.github/workflows/` does not exist. Pushing a branch that fails typecheck / lint / tests / RLS regressions / migrations is undetected. Vercel's default build only runs `next build`. Plus `next.config.ts` sets `eslint.ignoreDuringBuilds: true` so lint errors don't fail the build either. The highest-risk code paths (3-stage LLM, both Tekmetric webhooks, run-diagnostics 582-line aggregator) have **zero tests**, and the `test:e2e` script is broken (no playwright.config). 5 of the 10 BLOCKERs are this single gap. ~3 days to wire CI + write the missing tests.

Beyond the BLOCKERs, the highest-leverage non-launch-blocking action is **enabling Vercel BotID** on the chat + OTP endpoints. Today the only rate limit on OTP send is per-phone-per-hour (3); an attacker can mass-trigger 1000 distinct numbers per hour and cost the shop ~$4 in Telnyx SMS per pass. BotID is a 1-day fix.

The newly-built 3-stage diagnostic LLM (`diagnose-concern.ts`, refactored 2026-05-21) has the highest correctness risk per unit of test coverage: 1287 LoC, zero tests, just-refactored, drives every customer-facing recommendation. Adding ~10-15 unit tests around the prompt-assembly + structured-output gating + retry path is the single best test investment.

## 9. Positive findings — patterns to keep

The 6 agents identified an exceptionally consistent set of strong patterns. Do not regress these:

### scheduler-app

- **Universal `wrapAction` Sentry instrumentation** — every Server Action wrapped via `instrument-action.ts`, with auto-tagging of `wizard_action` + `chat_id`. New actions can't ship without instrumentation.
- **Zod everywhere** — every action's first line is `safeParse(args)`. Server Actions never trust input.
- **Server-side shop_id sourcing** — `SHOP_ID = 7476` is server-only; never URL/form/client.
- **PII scrubber is non-trivial + fail-closed** — 39-key blocklist + regex passes on every string leaf, dropping events on scrubber error.
- **Ephemeral session design** — 5-min idle → wipe-in-place via `hydrateSession`. Terminal states preserved for refresh.
- **Idempotency pre-flights on every Tekmetric write** — `submit-summary`, `submit-new-customer-info`, `submit-new-vehicle`. Prevents double-POST on retry/double-tap.
- **Race protection on rapid clicks** — `submit-date` + `submit-waiter-time` read `current_step` and no-op if advanced.
- **Hold release on Back** — `submit-back` releases `appointment_holds` so customer can't hold two slots simultaneously.
- **Post-confirm race protection in `mark-abandoned`** — snapshots `appointment_id` + skips abandon if booking already landed.
- **IdleTimer is solid** (per 2026-05-21 fix) — capture-phase listeners with mouse + visibilitychange + cleanup.
- **Two-layer error boundaries** — `app/error.tsx` + `app/global-error.tsx` both `captureException` and render `error.digest` (not `.message`).
- **Catch blocks are documented + intentional** — no silent error swallowing.

### Supabase functions

- **Pattern A bearer auth via `_shared/scheduler-auth.ts`** — constant-time compare, env-var fallback chain, structured diagnostic.
- **OAuth implementation** — SHA-256 hash at rest, PKCE S256, single-use codes with race-safe filter, rotating refresh tokens.
- **`_shared/sentry-edge.ts` PII scrubber** — 30+ key blocklist + regex + recursive value walker.
- **UUID confirmation token pattern + 6-char manual-review pattern** — exemplary; reference implementation for sensitive ops.
- **`appointments-sync` pagination de-duplication** via `Map<id, row>` last-write-wins.
- **Webhook handlers always return 200** after attempting work — correct semantics (no Tekmetric retry storm).
- **Resend `Idempotency-Key` per code** — prevents accidental double-send.

### Migrations / DB

- **Keytag system's 4-layer defense** (Pattern A + Pattern B + DB trigger + audit log) is the reference implementation for sensitive ops.
- **Money convention** — every `_cents` column is INTEGER/BIGINT; zero floats.
- **Soft-delete pattern correctly differentiated** — `deleted_at TIMESTAMPTZ` for sync-mirrored, `active BOOLEAN` for catalog.
- **Index coverage thorough** — every FK + GIN on every array col used in membership query.
- **All 21 SECURITY DEFINER functions** correctly set `search_path`.
- **`scheduler_error_log` + `scheduler_audit_log`** centralized observability tables; cron bodies retrofitted with EXCEPTION wraps in `20260516200000`.

### Integrations / observability

- **All secrets server-only** — no `NEXT_PUBLIC_*` leaks of server-side keys.
- **`beforeSend` scrubbers duplicated identically** in `sentry-edge.ts` + `sentry.server.config.ts`. Fail-closed.
- **Sentry Session Replay** at 10% baseline + 100% on error, with `maskAllText: true` + `maskAllInputs: true`.
- **AI Gateway model-fallback chain** configured (`providerOptions.gateway.models = [primary, fallback]`).
- **Cron exception handlers** wired on all 7 jobs (one broken — I-OBS-2).
- **`.env.*` blanket-ignored** in `.gitignore`. Live secrets never committed.
