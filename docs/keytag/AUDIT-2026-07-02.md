### H2 (F03) — `forceAssignAndPatch` treats `ro_already_has_tag` as success — PATCHes Tekmetric with a tag the DB did not assign, writes a false `force_assigned` audit row
`supabase/functions/_shared/tools/manual-review-tools.ts:622` · `manual-review-tools.ts:634` · `supabase/migrations/20260509215014_keytags_color_round_robin.sql:230`

The DRF/REG `use_prior_tag`/`use_different_tag` path errors only when `error_code !== 'ro_already_has_tag'`,
but `force_assign_keytag` returns that code **only when the RO holds a DIFFERENT tag** and performs no write
in that branch (the same-tag idempotent case returns NULL). When it fires, the code still PATCHes Tekmetric
with the requested wire value, writes a `force_assigned` audit row, and reports success. Reachable because
assigning a tag never auto-resolves the open review (M18). **Failure:** DRF open for an RO that meanwhile
got Y12 via the board; advisor resolves `use_prior_tag` (R5) → DB keeps Y12, Tekmetric shows R5, R5 stays
available for round-robin reuse, audit claims R5 was force-assigned — drift manufactured by the
drift-recovery tool, flip-flopped nightly by the reconcile re-patch. **Fix:** remove the exclusion (fail
with the current holder, like `dispatchArn`); never PATCH Tekmetric with a tag the DB write did not commit.
**Known:** new. *Verifier (upheld ×2):* RPC body verified (returns before any UPDATE in that branch);
`dispatchArn`'s `track_tag` fails on any error_code, proving the exclusion is an anomaly; auto-resolve
fires only at release sites, so open-review-plus-board-tag is a normal state.
