## Locked decisions

| # | Decision | Rationale |
|---|---|---|
| D1 | One private bucket `vehicle-docs`; architecture supports more buckets later (profile rows carry `bucket`). | Chris: "same bucket, just different titles / email / scan profile." |
| D2 | Object path is **constructed server-side only** (edge fn or trigger — never by the shop PC): `{shop_id}/{profile_key}/{channel}/{YYYY}/{MM}/{ts}_{sha8}.{ext}` (`channel` = `scan` \| `email`). Basename is **opaque** — the original filename lives ONLY in the DB row (filenames carry PII + parsing hazards; keys leak into logs/URLs). `ts`+key are minted once per job and persisted (agent job ledger / attachment row) so retries reuse the SAME key — the key is an idempotency token, not a fingerprint. Full `sha256` stored in the row; `sha8` in the key is uniqueness seasoning only. | Cross-verify: client-authored keys were an authz hole + PII leak + retry-duplication source. |
| D3 | Row creation is **explicit at upload time** (the agent-gateway fn and the email fn both insert `document_intake_files` rows themselves, with rich metadata), PLUS a **catch-all AFTER INSERT trigger** on `storage.objects` (bucket-filtered) that upserts a bare row `ON CONFLICT (object_path) DO NOTHING` — belt for objects that arrive any other way. Trigger fn is **`SECURITY DEFINER`, owned by `postgres`, `SET search_path = public`, `REVOKE EXECUTE FROM public`** (it must write deny-all-RLS tables regardless of the storage-API role that fired it — verified requirement, both reviewers). Body wrapped `EXCEPTION WHEN OTHERS` → `document_intake_error_log` (never blocks an upload). A **reconciliation step in the daily cron** diffs `storage.objects` ↔ `document_intake_files` and registers/flags orphans — trigger-swallowed errors cannot silently strand files. | Explicit registration = correct data; trigger + reconciliation = nothing invisible, ever. |
| D4 | Routing config lives in DB: `document_intake_profiles` (key PK, shop_id, label, bucket, intake config, active) + **`document_intake_mailboxes`** child (`address` UNIQUE on `lower(address)`, profile_key FK) so one mailbox can never route ambiguously. (v1 drops the redundant `prefix` column — the path segment IS `profile_key`.) The agent does NOT duplicate this config: it fetches its watch-folder→profile map from the agent-gateway fn (D5) at startup + hourly, caching last-good locally. Seeded: `inspection_docs` ← inspection@, `loaner_insurance` ← loaner@. | `shop-agnostic.md` (config in DB); kills the config-drift + ambiguous-routing findings. |
| D5 | **The shop PC holds NO storage credentials.** Agent auth = single `AGENT_TOKEN` secret to a new **`document-intake-agent` edge fn** (Pattern A-style bearer), which serves: `GET config` (profiles/folders), `POST request-upload` (validates profile, mints the server-side path, creates the `pending` row, returns a `createSignedUploadUrl` token), `POST confirm` (verifies the object exists + sha256/size match → row `ready`), `POST heartbeat`. The signed upload URL is single-path, short-lived, upload-only — the PC can never read, list, overwrite, or delete anything, and cannot author another shop's/profile's path. | Replaces v1's project-wide S3 keys (both reviewers: unacceptable blast radius + path-as-authz hole). Also gives the agent DB-truth config + confirm-after-upload ordering for free. |
| D6 | Agent runs as a **Windows Service** (node-windows; winsw-based, active) — auto-start at boot, no login needed. Runbook sets `powercfg` never-sleep + **Windows Time (NTP) health** (signed URLs + TLS are clock-sensitive). NSSM stale / WinSW limbo; Task Scheduler documented as fallback. | Chris's hard requirement: works any time the PC is on, logged out or not. |
| D7 | **Email channel = Microsoft Graph change notifications on two M365 shared mailboxes — `inspection@` + `loaner@jeffsautomotive.com`** (Chris: "prefer our domain"). No subdomain, no MX change, no forwarding; mail never leaves the tenant until we pull attachments. Entra app registration + client-credentials daemon (no user, no MFA, nothing to sign out); subscription per mailbox on Inbox messages (`changeType: created`) with `notificationUrl` AND **`lifecycleNotificationUrl`** (a separate subscription property — lifecycle events do NOT go to the resource URL) + **per-subscription random `clientState`**. All Graph reads use **immutable IDs** (`Prefer: IdType="ImmutableId"`) so moved messages stay fetchable. **Mailbox scoping:** app gets NO tenant-wide Entra mail grant; access comes solely from Exchange **RBAC for Applications** (`New-ServicePrincipal` + `New-ManagementScope` on a mail-enabled security group holding exactly these two + `New-ManagementRoleAssignment -Role "Application Mail.Read" -CustomResourceScope` + `Test-ServicePrincipalAuthorization` proving both in-scope AND chris@ out-of-scope) — plus an ops-time **end-to-end Graph test** (subscribe + read allowed mailbox succeeds; unscoped mailbox 403s) since the cmdlet alone doesn't exercise Graph. Resend = sending only (Receiving disabled; stale 2026-03 webhook deleted 2026-07-21 with Chris's approval). Fallback design (Resend inbound on subdomain) documented in research.md addendum. | Root-domain addresses exactly as Chris wants; least privilege proven twice; verified docs in research addendum. |
| D8 | **Durable-job email pipeline** (reworked per cross-verify — ACK-before-work is not durable, dedup ≠ retry): (a) subscription **validation handshake** — echo `validationToken` (`text/plain 200`); (b) notifications — resolve the stored subscription row by `subscriptionId` (unknown/expired → reject), constant-time verify that subscription's `clientState` (fail-closed; invalid traffic is **rate-capped + Sentry-sampled**, not 1:1 events) → upsert `graph_mail_events` rows (`status='pending'`, dedup on `(mailbox, immutable message id)`) → **202 after the store** → best-effort immediate processing via `EdgeRuntime.waitUntil` (documented Supabase mechanism for post-response work). (c) **Daily cron mode** (Pattern A bearer, pg_cron; **advisory-lock serialized** so overlapping runs can't race): **renew** subscriptions requesting **≤2.5-day expirations** (docs say mail max is 10,080 min; Gemini says 4,230 — requesting 2.5 days is valid under BOTH, and daily renewal still leaves ≥1.5 days slack), recreate expired/removed, honor lifecycle events; **sweep** each mailbox's Inbox AND Junk over a **rolling 7-day window** (no watermark — the events-table dedup makes re-listing idempotent; fixed overlap kills every watermark race/loss case), page `@odata.nextLink` fully; **drain** all `pending`/`retryable` events + attachments (attempts, backoff, `next_retry_at`); **reconcile** storage↔rows (D3) and run watchdog checks (D13). Event state machine: `pending → processing → completed | retryable(attempts++) | failed`; **per-attachment child rows** (`graph_mail_attachments`: immutable msg id + attachment id, status, skip_reason, object_path, attempts) so a 3-attachment message can succeed partially and retry exactly the missing one. | Webhook = latency optimization; the cron is the guarantee. Every reviewer failure mode (killed isolate, transient 429/404, partial attachment success, watermark gaps, moved messages, junked mail) lands in a durable state that the next cron drains. |
| D9 | Accepted types: `application/pdf`, `image/jpeg`, `image/png`, `image/heic`/`heif` — validated by **magic bytes**, not sender-declared MIME/extension, on both channels; per-file (40MB) + per-message caps; **inline images (`isInline`) and non-file attachments (`itemAttachment`/`referenceAttachment`) skipped** (signature logos are not documents; skips recorded with reason). Off-type/oversize → attachment row `skipped`. Rows are `pending` until validation passes → `ready`; failures → `rejected` (never silently dropped). HEIC pass-through is a **documented UI debt**: browsers can't render it — the future viewer converts or downloads (noted for the inspection/loaner apps). Malware scanning: NOT in v1 (Chris call; mitigations = magic-byte gate + private bucket + future viewer serves `Content-Disposition: attachment`); candidate fast-follow. | Sender-controlled metadata is untrusted; validation is ours. |
| D10 | v1 extracts NOTHING from documents (no OCR, no card fields in DB — PII stays in the private bucket; extraction is its own feature). `document_intake_files.status`: `pending → ready → linked` + `rejected` / `failed`; unrouted mail = `profile_key NULL` under `{shop_id}/unrouted/email/…` (kept, queryable). `linked_ref jsonb` is v1's claim marker for the future apps, with a documented commitment: when the inspection app lands, claiming becomes an **atomic shop-scoped RPC** (proper link table if many-to-many emerges) — v1 has no consumers, so nothing can race it yet. | Keep v1 small; the claim contract is written down before any consumer exists. |
| D11 | All new tables: deny-all RLS (service-role only), `shop_id integer NOT NULL` (Tekmetric id; Jeff's = 7476), TIMESTAMPTZ, uuid PKs — back-office conventions. The D3 trigger's SECURITY DEFINER path is **integration-tested under the real storage-API role** (pgTAP sets the role, inserts a storage.objects row, asserts the intake row by COUNT). | `cross-module-anchors.md`; the RLS-vs-trigger conflict both reviewers flagged is closed by test, not assumption. |
| D12 | Shop-PC file lifecycle, hardened: watch `C:\Scans\{folder}` (the ONLY SMB-shared tree; **archives live OUTSIDE the share** at `C:\ScanAgent\archive\{uploaded,failed}\{profile}` with SYSTEM+Administrators-only ACLs — retained cards are never browsable on the LAN). Pipeline per file: chokidar event OR periodic directory re-scan (notifications get lost; polling reconciles) → **stability gate** (size+mtime unchanged 10s + exclusive-open probe + min-size sanity; scanners don't temp-rename) → **atomic claim** (rename to UUID staging name inside `C:\ScanAgent\work` — kills source-name collisions + double-processing) → append to the **persistent local job ledger** (JSONL: staging name, original name, sha256, profile, minted object path, attempts, state) → request-upload → PUT → confirm → archive. Ledger replay on service start resumes any state. Retries: exponential backoff capped at 30 min, forever, **with a free-disk floor (pause + alarm below 5GB) and a failed-count alarm (>25 pending → Sentry)**. `_uploaded` purged after 30 days; `_failed` never auto-purged. Cross-drive moves via copy+fsync+unlink (rename isn't atomic across volumes). SMB share hardening in runbook: dedicated local `scanner` account, write-only share ACL, SMB3+signing required, agent ignores reparse points + non-regular files. | Every reviewer scan-side finding (partial writes, name collisions, lost events, PII-browsable archives, unbounded retries, share as attack surface) has a concrete mechanism. |
| D13 | **Sentry alerting v1 — active failures AND silent stalls** (Chris 2026-07-21). Tag `module: document-intake` on surfaces `scan-agent`, `document-intake-email`, `document-intake-agent`. Agent: Sentry required (DSN of `jeffs-app-v2-supabase`), captureException after 3rd consecutive failure + the D12 alarms. Email fn: captures on processing failures (sampled for unauthenticated junk traffic). **Watchdog promoted into v1** (was deferred; the cron + gateway fn make it ~free and reviewers correctly called silent-stop the deadliest mode): the daily cron checks — agent heartbeat age (>2h during shop hours), last-scan and last-email intake age vs profile expectations, Graph subscription expiry proximity, pending/retryable backlog age, `document_intake_error_log` new rows, reconciliation orphan count — each breach → `Sentry.captureMessage` (warning/error). **Alert rule** on `jeffs-app-v2-supabase`: "event where `module=document-intake` → email Chris" — provisioned **idempotently** (look up by name before POST) via the Internal Integration API (`sentry_api_and_cli.md` guardrails) or dashboard. E2E smoke includes forcing a failure to see the email arrive. | "No silent failures" now includes "no silent silence." |
